## Cross Site Scripting = xss 跨站脚本攻击

### 反射性xss 
- post请求提交以后 等到后端把数据返给前端用来展示的时候,如果有<\script><\/scipt>里面的代码就会被浏览器解析并且直接执行 
> 反射性,诱导用户自己点开.
> 谁点开,谁的数据受害
解决办法: 
前端: 
1. 前端可以正则处理里面是否含有< script>,有就处理
后端: 
1. 后端把提交代码encodeinComponent处理解析返回
2. httponly

### dom-based
- 修改属性,插入内容(给图片的src直接赋值) , document.write 都会执行传入的脚本
解决办法: 
前端: 
- encodeURI包裹传入的值,用转义
- 用正则匹配</\script>

### xss的存储型攻击
<!-- 之前微博会以自己的身份发送一条恶意微博 -->
- 评论功能
每次把评论功能的数据存储到服务器上,所有访问的人都会被攻击,比xss的反射性和dom-based型攻击范围都大
解决办法: 
前端: 
1. 客户端传给服务端的时候就用正则过滤一下,但是接口还是会被拦截或者仿造接口. 也可以用转义 尖叫号lt gt等 字符串替换
2. 输入的list表也要过滤一下<\script> 
后端: 
服务器端做过滤

### xss解决方法总结

前端: 
- 字符串替换转义 /'/g.replace('%amp')
- 字符串正则匹配 /<\script><\/script>/






## csrf 跨站伪造请求
- Cross-site request forgery = csrf 跨站请求伪造 登录人家网站,伪造用户cookie

初始化有用户信息
钓鱼网站,有用户信息
给用户一个吸引他的网站 
需要用户点进这个网站
网站里访问一个新的网站(用ifreame访问)

前提必须是登录过: 
用户访问正规网站后,
本地有cookie,
然后诱导用户点击坑人网站,
会用一个隐藏的iframe网站里提价一个表单
自动会调取本地cookie用正规网站的cookie(并没有拿到cookie,只是发请求,会自动携带本地正规网站cookie)
然后访问正规网站,按照用户提交的表单,默认重新提交一份,就会把钱盗走

防御方法三种:
1. 添加验证码(体验不好) 伪造的请求是没法填写验证码的
   1. 后台判定用户信息的时候就给一个验证码,前端展示图片
   2. 用户提交的时候多一个字段,就是后端自己存储真实内容但是只把图片传给前端的验证码
   3. 如果提交的时候后端判断验证码不对,就无法登陆.
2. 判断来源,假如是3001接口发送的3000的接口,也是伪造的
   1. Header头里面判断Referer字段来源是否为某个值
   2. 但是这个值可以伪造,以为header里可以设置

```javascript
let referer = req.headers['refere'];
if(referer.includes('xxx')){
    // 如果是指定路径,就可以使用. 感觉类似于白名单
}

```
3. token
> 类似于验证码,但是用户的感知不强烈. 也是后台存储一个验证码,前端每次请求携带token
   1. 前后台约定好一个token(token的组成就是两人约定好把cookie值添加一个特定的字段,用来后台判定)
   2. 每次发送请求的时候判断是否是约定好的token来判断,但是也是基于其他页面拿不到cookie
